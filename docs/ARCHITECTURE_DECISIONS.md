# 架构决策记录

## ADR-0001：采用轮次驱动执行协议

- 状态：ACCEPTED
- 日期：2026-04-19

### 决策

当前仓库后续开发按 `R00` ~ `R10` 的轮次推进，并以 `docs/EXECUTION_LEDGER.md` 作为唯一执行账本。

### 理由

- 当前仓库已有较多实现，不能再用“想到哪改到哪”的方式推进
- 业务文档与代码实现存在差异，必须通过分轮次门禁控制风险
- 需要可追溯地记录每轮输入、改动、命令和验收结果

## ADR-0002：保持模块化单体，不改成微服务

- 状态：ACCEPTED
- 日期：2026-04-19

### 决策

继续采用当前 monorepo + 模块化单体架构：

- 前端：`apps/web`
- 后端：`apps/api`
- 共享包：`packages/shared`

### 理由

- 当前仓库已按此方向落地
- 与项目 AGENTS 约束一致
- 现阶段问题是流程对齐和工程收口，不是服务拆分

## ADR-0003：流程流转继续由后端控制

- 状态：ACCEPTED
- 日期：2026-04-19

### 决策

流程模板、节点生成、退回、新轮次、SLA 和周期任务都由后端控制；前端仅提交动作和展示状态。

### 理由

- 与正式业务文档一致
- 与当前仓库既有实现一致
- 能避免前端绕过状态机造成脏状态

## ADR-0004：当前仓库按“增量对齐”而非“重写”推进

- 状态：ACCEPTED
- 日期：2026-04-19

### 决策

后续轮次采用增量改造，不推倒重写现有实现。

### 理由

- 当前仓库已经具备完整工程底座和多个业务模块
- 大规模重写会放大交付风险
- 用户要求按轮次、按门禁推进

## ADR-0005：把正式文档作为目标口径，把当前实现作为迁移起点

- 状态：ACCEPTED
- 日期：2026-04-19

### 决策

三份正式文档是目标口径；当前仓库是迁移起点。两者不一致时，不隐式替换，而是在后续轮次里显式修正。

### 已识别差异

- 当前流程末端尚未形成“第 17 步月度周期 + 第 18 步退出治理”的正式模型
- 当前 `computeDueAt()` 仍是占位算法，不满足正式 SLA 规则
- 当前包名仍保留历史命名 `feishu-timeline-plugin`

### 影响

- `R01` 不处理业务语义对齐，只打通工程底座
- `R02` 和 `R03` 才开始模型和流程规则对齐
