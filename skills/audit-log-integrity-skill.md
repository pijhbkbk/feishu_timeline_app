# Audit Log Integrity Skill

本技能面向 Audit Log Agent、API Agent、Workflow Agent 和 ColorDev Orchestrator Agent，用于规范“轻卡新颜色开发项目管理系统”中关键写操作的审计留痕、追溯链路、排错依据与验收方式。它只服务当前仓库的主链路节点、评审节点、附件、通知和颜色退出等业务场景，不服务通用审计平台设计。

本技能的底线有两条：

- 审计日志必须落库，不能只打控制台日志。
- 项目关闭前，主链路节点、评审记录、附件、日志必须可追溯。

## 1. Skill 目标

- 为本仓库关键写操作建立统一、可落库、可追溯、可排错的审计规范。
- 明确 `audit_logs` 与 `workflow_transitions`、`review_records`、`notifications` 的职责分工。
- 约束“谁触发、何时触发、前后状态、关联对象、结果如何”的记录方式。
- 规范幂等请求、重复提交、重复写入下的审计处理方式。
- 让审计日志成为 workflow、review、attachments、notifications 排错与回溯的可靠证据链。

## 2. 适用场景

- 项目立项、更新、关闭、颜色退出等项目级关键写操作。
- workflow 主链路推进、并行节点创建、退回、撤回、重提。
- 驾驶室评审、一致性评审、目视色差评审的通过、驳回、补充、重提。
- 附件上传结果落库、附件与业务对象绑定、附件逻辑删除。
- 通知写入 `notifications`、投递状态变化、重试状态变化。
- 权限敏感或状态敏感的关键写接口。
- 需要对线上问题进行回放、排错、回溯的流程型缺陷分析。

## 3. 不适用场景

- 纯前端展示、纯 UI 文案、纯视觉样式改动。
- 不改变任何业务状态的只读查询。
- 临时调试输出、开发期本地 console 观察。
- 与业务对象无关的低价值噪声事件。
- 试图把当前仓库扩展成通用 SIEM、全链路观测平台或审计白皮书。

## 4. 审计日志在当前系统中的角色

在当前系统中，`audit_logs` 的角色不是“补充说明”，而是关键写操作的事实留痕。它服务三个目的：

- 合规追溯：谁在什么时间对哪个对象做了什么动作。
- 业务排错：为什么某个项目进入了某个节点，为什么某次评审退回了，为什么某个附件关系变化了。
- 交付验收：证明关键操作已经具备可追溯性，满足仓库 Done 要求。

它与系统其他记录协同：

- `workflow_transitions` 记录流程怎么流转。
- `review_records` 记录评审为什么放行或拦截。
- `notifications` 记录通知是否写入和发送状态。
- `audit_logs` 记录是谁在什么时候触发了这些关键写操作，以及动作结果如何。

## 5. 哪些操作必须记录 `audit_logs`

至少以下操作必须记录审计日志：

- 项目创建、项目关闭、颜色退出。
- 主链路节点推进、退回、撤回、重提。
- 并行节点创建、关闭、重开。
- 评审通过、驳回、补充资料、重提。
- 附件元数据入库、附件与项目/评审/流程对象绑定、附件逻辑删除。
- 会改变 `projects.current_node`、状态、负责人、关键日期、优先级的写操作。
- 会改变 `workflow_tasks` 状态、负责人、截止时间的关键写操作。
- 会写入 `notifications` 或改变其发送状态、重试状态的重要操作。
- 会改变权限敏感对象的关键写接口，例如只有 `project_manager`、`reviewer`、`admin` 等可执行的高风险动作。

判断原则：

- 只要动作会改变业务状态、门禁结果、责任归属或追溯链，就应写 `audit_logs`。
- 只要动作未来可能需要回答“是谁做的、为什么做、做完结果是什么”，就应写 `audit_logs`。

## 6. 审计日志与 `workflow_transitions` 的分工

| 对象 | 主要回答的问题 | 典型内容 |
| --- | --- | --- |
| `workflow_transitions` | 流程从哪里到了哪里 | 来源节点、目标节点、流转动作、时间点 |
| `audit_logs` | 谁触发了这次关键动作，结果怎样 | 操作者、角色、对象、动作、前后状态、结果、上下文 |

分工原则：

- 不能用 `workflow_transitions` 代替 `audit_logs`。
- 不能用 `audit_logs` 代替 `workflow_transitions`。
- 对于流程型关键动作，通常两者都要有：
  - `workflow_transitions` 记录节点变化事实。
  - `audit_logs` 记录触发人、触发原因、动作结果和上下文。
- 若只有流转没有审计，无法追责。
- 若只有审计没有流转，无法还原流程路径。

## 7. 建议记录字段

建议 `audit_logs` 至少覆盖以下信息：

- `action_type`：动作类型，例如项目关闭、评审通过、节点退回、附件绑定。
- `object_type`：对象类型，例如 `project`、`workflow_task`、`review_record`、`attachment`、`notification`。
- `object_id`：被操作对象主键。
- `project_id`：所属项目 ID，便于跨对象排查。
- `operator_id`：执行人 ID。
- `operator_name`：执行人展示名，若当前系统有此字段来源。
- `operator_role`：执行动作时的业务角色。
- `trigger_source`：触发来源，例如 API、系统自动流转、定时任务、异步补偿。
- `occurred_at`：发生时间。
- `before_state`：动作前关键状态。
- `after_state`：动作后关键状态。
- `result`：成功、拒绝、幂等命中、失败。
- `reason`：驳回原因、退回原因、补充说明等。
- `related_type` / `related_id`：关联对象，例如某次评审、某条通知、某个附件。
- `request_id` 或幂等键：用于串联一次请求链路。

字段命名可按现有表结构落地，但语义层面不应缺位。

## 8. “谁触发、何时触发、前后状态、关联对象、结果如何”的记录原则

每条高价值审计日志都应尽量回答以下五个问题：

1. 谁触发：哪个用户或哪个系统动作发起了操作？
2. 何时触发：动作发生的精确时间是什么？
3. 前后状态：动作前后关键状态分别是什么？
4. 关联对象：它影响了哪个项目、哪个节点、哪次评审、哪个附件或哪条通知？
5. 结果如何：成功了、被拒绝了、幂等命中了，还是失败了？

落地原则：

- 记录前后状态时，不求全量对象快照，但必须包含足够排错的关键字段。
- 对流程动作，至少包含来源节点和目标节点。
- 对评审动作，至少包含评审类型、结论和关联项目。
- 对附件动作，至少包含附件对象、绑定对象和操作类型。
- 对通知动作，至少包含通知对象、发送状态变化和触发来源。

## 9. 幂等请求与重复写入的处理原则

- 关键写接口必须幂等，审计也必须考虑幂等语义。
- 幂等命中时，不应伪造一条“成功完成新动作”的审计日志。
- 推荐做法是：
  - 若请求首次成功，写一条成功审计日志。
  - 若后续重复请求命中幂等，写明 `result=idempotent_hit` 或等价语义，或者在既有日志链路中可追溯地标记重复命中。
- 不要因为防重而完全失去重复请求的排查能力。
- 不要因为重复请求而制造多条看似独立成功的关键动作日志。

判断重点：

- 审计要忠实记录“发生了什么”，而不是“看起来应该发生什么”。
- 对重复提交、前端多次点击、消息重试造成的调用，要能区分首次执行与后续命中。

## 10. 审计日志与排错、回放、回溯的关系

审计日志不是只为“留档”。在当前仓库中，它还必须支持：

- 排错：定位某个项目为什么进入当前节点，为什么某次评审没有放行，为什么通知重复触发。
- 回放：按时间顺序重建关键写动作链，辅助人工还原问题现场。
- 回溯：在项目关闭前查看主链路节点、评审记录、附件、日志是否完整可追溯。

与其他记录形成链路时，建议遵循：

- 先用 `project_id` 聚合项目范围。
- 再用 `workflow_transitions` 看流程走向。
- 用 `review_records` 看门禁结论。
- 用 `notifications` 看提醒是否写入与投递。
- 用 `audit_logs` 串起“是谁触发了这些变化”。

若 `audit_logs` 缺失，排错通常会卡在“知道状态变了，但不知道谁改的、为何改的”。

## 14. 常见错误与禁止事项

- 只打 console 日志，不落库。
- 只记录“成功”，不记录被拒绝、幂等命中或失败结果。
- 只写 `workflow_transitions`，不写 `audit_logs`。
- 只写评审记录，不写审计日志。
- 审计日志缺少操作者、项目、对象、时间等关键上下文。
- 把所有低价值事件都写进审计，导致关键动作淹没。
- 在 Controller 或前端页面里临时拼装审计字段，导致语义漂移。
- 覆盖或删除历史审计记录。
- 无法区分系统自动动作和人工触发动作。
- 修改颜色退出、项目关闭等高风险动作却不补审计。

## 16. 示例

### 16.1 评审通过操作如何留痕

示例场景：`COLOR_CONSISTENCY_REVIEW` 通过，项目从一致性评审进入 `MASS_PRODUCTION_PLAN`。

应记录的内容：

- `review_records` 中有本次评审通过记录。
- `workflow_transitions` 中有 `COLOR_CONSISTENCY_REVIEW -> MASS_PRODUCTION_PLAN` 的流转记录。
- `audit_logs` 中有一条“评审通过”动作日志，至少包含：
  - 操作者和角色，例如 `reviewer`
  - 关联项目 ID
  - 当前评审节点
  - 前后状态或前后节点
  - 结果为成功
  - 请求标识或幂等键

排错价值：

- 可以同时回答“流程怎么推进的”和“是谁批准推进的”。

### 16.2 颜色退出操作如何留痕

示例场景：项目进入 `PROJECT_CLOSED`，执行颜色退出。

应记录的内容：

- 主链路关闭或颜色退出涉及的状态变化。
- 关联的 `color_exits` 或项目关闭对象上下文。
- 操作者、角色、触发时间、关闭原因或退出说明。
- 若同时有通知写入，应能从审计中找到通知触发来源。

特别要求：

- 颜色退出属于高风险、高追溯要求动作，不能只记录节点变化，必须记录人、因、对象、结果。

### 16.3 错误示例：只打 console 日志

错误做法：

- 在 API 或 Service 中写 `console.log("review approved")`。
- 数据库里没有任何 `audit_logs` 记录。
- 线上问题只能看到流程状态变了，但无法查到是谁、何时、为什么触发。

为什么错误：

- console 日志不等于审计日志。
- console 不具备稳定查询、对象关联、项目维度检索和长期追溯能力。
- 它无法满足“项目关闭前，主链路节点、评审记录、附件、日志必须可追溯”的仓库要求。

正确结论：

- 关键写操作必须把审计信息落到 `audit_logs`，console 只能作为开发期辅助，不能作为交付结果。
