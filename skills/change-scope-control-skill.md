# Change Scope Control Skill

本技能面向 ColorDev Orchestrator Agent、Regression Guard Agent、Bug Fix Agent，以及所有下级执行 Agent，用于规范“轻卡新颜色开发项目管理系统”中的改动范围控制方式，防止任务越界、文件扩散、无关模块被误改，以及 AI 自由发挥导致仓库失控。它只服务当前仓库的模块化单体、主链路优先和有边界推进原则，不服务抽象项目管理口号。

本技能的四条底线：

- 有边界地推进。
- 非目标范围必须显式写出。
- 未经授权，不得跨到无关模块。
- Diff 必须通过范围门禁后，才能进入验收。

## 1. Skill 目标

- 为本仓库建立明确、可执行的改动范围控制规则。
- 把“单节点、单目标、单轮交付”落成可检查的工程实践。
- 防止一次任务被扩展成多目标改造、无关重构或全仓自由发挥。
- 为 Orchestrator、Regression Guard、Bug Fix 和执行 Agent 提供统一的范围门禁。
- 让每一轮改动都能回答“为什么改这些文件，而不是更多”。

## 2. 适用场景

- 新增一个业务节点或调整一个已有节点。
- 修改一条退回规则或一类评审门禁。
- 修复单个 workflow / review / notification / audit 缺陷。
- 新增单类通知、单类审计或单类 DTO 合同变更。
- 需要多个 Agent 协同，但本轮目标明确、边界可收敛的任务。
- 任何需要审查“文件改动是否超出任务目标”的执行回合。

## 3. 不适用场景

- 已明确授权的大型重构、目录迁移或多阶段架构升级。
- 仓库初始化、空白功能搭建、一次性大规模迁移。
- 用户明确要求全局整理、统一风格、全量重命名。
- 未给出边界、且确实需要先做专门规划再执行的复杂项目。

即便在不适用场景下，本技能的“写清目标、范围、非目标”仍然有参考价值。

## 4. 为什么改动范围控制对本仓库重要

本仓库不是通用脚手架，而是“轻卡新颜色开发项目管理系统”MVP，具备以下特点：

- 主链路清晰，但 workflow、review、RBAC、notifications、audit、attachments、Prisma 会联动。
- 规则集中在后端，前端只负责展示与交互。
- 每次小改动都可能影响 `workflow_transitions`、`review_records`、`audit_logs`、`notifications` 等追溯链。
- 仓库已明确禁止为了“通用性”提前引入复杂引擎或无关抽象。

如果没有改动范围控制，最容易出现的问题是：

- 修一个 review 问题，顺手改 workflow、dashboard、shared、通知文案和无关页面。
- 修一个 API bug，顺手重构 auth、DTO 命名和目录结构。
- 为了让测试通过，大面积改断言或放宽校验。

这类扩散会直接破坏：

- 可验证性
- 可回滚性
- 追溯性
- 交付节奏

## 5. 单节点、单目标、单轮交付的原则

### 5.1 单节点

- 一轮任务优先围绕一个主链路节点、一个评审节点、一个通知事件或一个缺陷触发点展开。
- 如果问题跨节点，必须说明哪个节点是主修改对象，哪个只是联动影响。

### 5.2 单目标

- 一轮只解决一个明确业务目标，例如：
  - “调整一致性评审退回规则”
  - “修复重复待评审通知”
  - “补齐颜色退出审计日志”
- 不允许把多个无关诉求打包成“顺手一起做”。

### 5.3 单轮交付

- 一轮交付必须有清晰开始、清晰范围、清晰验收和清晰回滚点。
- 若任务在执行中扩大，必须停下来回到计划层，而不是边做边膨胀。

## 6. 如何定义本轮目标、范围、非目标

建议每一轮任务都显式写出：

```markdown
## 本轮目标
- 只解决什么问题？

## 本轮范围
- 允许修改哪些模块、哪些文件类型、哪些节点、哪些表、哪些页面？

## 本轮非目标
- 明确不做什么？
- 哪些相邻问题即使看到了也不顺手处理？
```

### 6.1 本轮目标

目标必须是业务可理解、工程可落地的单一事项，不能写成：

- “整体优化流程”
- “顺手统一一下”
- “把这里也规范掉”

### 6.2 本轮范围

范围建议至少写到以下粒度：

- 模块级：`apps/api/workflows`、`apps/web`、`packages/shared`
- 业务级：`COLOR_CONSISTENCY_REVIEW`、`notifications`
- 数据级：`review_records`、`workflow_transitions`

### 6.3 本轮非目标

非目标必须显式写出，例如：

- 不重构 dashboard
- 不调整 auth
- 不统一全仓 DTO 命名
- 不修复与本次通知问题无关的历史 UI 问题

没有非目标，通常意味着边界不清。

## 7. 如何判断“必要改动”与“越界改动”

### 7.1 必要改动

满足以下任一条件，可视为必要改动：

- 直接修复目标问题的根因代码。
- 为了让主修复成立而必须同步的 DTO、shared enum、测试断言。
- 为了保持 workflow / review / notification / audit 联动一致而必须补齐的小范围改动。
- 为了通过仓库既有检查链路所必需的配套修复。

### 7.2 越界改动

满足以下任一条件，应视为越界：

- 与本轮目标没有直接因果关系。
- 只是“看起来可以顺手优化”。
- 只因为当前文件正好打开了，就顺手一起改。
- 会扩散到新的业务目标、新的模块所有权或新的验收维度。
- 改动后无法用“本轮目标”解释其必要性。

判断问题模板：

- 这处修改是否直接服务本轮目标？
- 不改它，本轮目标是否无法成立？
- 如果必须改，是否已在本轮范围里写清？

只要连续两个回答是否定的，就应停止修改。

## 8. 常见越界类型

- 修 review，顺手改 dashboard。
- 修 API bug，顺手重构 auth。
- 修通知重复，顺手统一消息文案和 UI 样式。
- 修后端状态流转，顺手改前端本地状态管理方式。
- 为了“整洁”，顺手改无关 DTO、shared constants、目录命名。
- 修一个失败测试，顺手放宽多个无关测试断言。
- 修 workflow 规则，顺手把通用引擎雏形搭起来。

## 9. Diff Gate 的建议规则

Diff Gate 是本技能最重要的工程门禁。建议至少执行以下规则：

### 9.1 目标对齐规则

- 每个被改文件都能回答“它为什么服务本轮目标”。
- 若某个文件无法一眼解释其必要性，应默认进入人工复审或打回。

### 9.2 非目标对齐规则

- 任何落在非目标范围内的改动默认打回。
- 若执行中发现非目标必须改，必须先回到 Plan 重新定义范围。

### 9.3 文件数规则

- 单轮优先控制在最小必要文件集。
- 文件数异常增多时，应触发范围复核，而不是默认“问题比较复杂”。

### 9.4 模块跨度规则

- 单轮跨模块联动必须能说明主修复链路。
- 若同时触达 `apps/web`、`apps/api`、`packages/shared`、`prisma`，必须说明每一层的必要性。

### 9.5 语义漂移规则

- 如果 diff 中出现大规模重命名、格式化噪声、无关文案改动、无关注释调整，应优先剔除。

### 9.6 规则总结

推荐用以下问题做 Diff Gate：

```markdown
1. 这个文件为什么必须改？
2. 不改这个文件，本轮目标是否仍能完成？
3. 这处修改是否落在本轮范围内？
4. 这处修改是否误触非目标？
5. 这处修改会不会引入新的独立验收面？
```

## 10. 何时应打回重做

出现以下情况，应打回重做：

- 本轮目标与实际 diff 不一致。
- diff 中出现非目标模块改动，且未提前声明。
- 修复一个 bug，却引入大规模无关重构。
- 范围已扩展到需要新增新的验收维度。
- 执行人无法解释某些改动为何必要。
- 需要跨模块联动，但本轮计划没有写清依赖和门禁。
- 回归验证发现改动面已经超出原定可控边界。

打回不等于失败，而是为了恢复“有边界地推进”。

## 11. 何时允许跨模块联动

跨模块联动不是天然越界，但必须满足全部条件：

- 本轮目标本身就是跨层联动问题。
- 每一层改动都有明确必要性。
- 非目标仍然清晰。
- 回归验证范围仍可控。
- Orchestrator 已明确写出影响面与门禁。

典型允许场景：

- API DTO 变更导致前端页面消费必须同步。
- workflow 退回规则修复导致通知幂等键和审计埋点必须同步。
- shared enum 新增导致前后端分支必须同步更新。

即便允许跨模块，也不等于允许自由扩散。

## 12. 设计检查清单

- [ ] 本轮目标是否是单一、明确、可验收的？
- [ ] 本轮范围是否已写到模块、节点、数据对象粒度？
- [ ] 本轮非目标是否显式列出？
- [ ] 是否识别出可能的跨模块联动点？
- [ ] 每个联动点是否都有必要性说明？
- [ ] 是否定义了 Diff Gate 检查问题？
- [ ] 是否定义了打回条件？
- [ ] 是否有清晰的回滚点和验收边界？

## 13. 编码检查清单

- [ ] 只修改本轮范围内的文件。
- [ ] 每个被改文件都能解释其必要性。
- [ ] 未触碰非目标模块。
- [ ] 未顺手做无关命名、无关格式化、无关重构。
- [ ] 若出现必须扩展范围的情况，已停止并回到计划层。
- [ ] workflow / review / notification / audit 的联动修复仍然保持最小必要集。
- [ ] 未为了“整洁”或“通用性”引入额外抽象。

## 14. 验收检查清单

- [ ] diff 与本轮目标一致。
- [ ] diff 未包含非目标改动。
- [ ] 改动文件集保持最小必要规模。
- [ ] 跨模块改动均有必要性说明。
- [ ] 验证结果覆盖了本轮范围，而不是无限扩展。
- [ ] 可以清楚说明“本轮做了什么、没做什么”。
- [ ] 若需要后续任务，已经被显式留到下一轮，而不是夹带在本轮里。

## 15. 常见错误与禁止事项

- 不写非目标。
- 发现相邻问题就顺手一起修。
- 修 review 顺手改 dashboard。
- 修 API 顺手重构 auth。
- 看到旧代码不顺眼就顺手统一风格。
- 以“反正都在这个文件里”为理由夹带无关修改。
- 先写一大堆 diff，再回头倒推范围解释。
- 把一次局部修复做成潜在平台化改造。

- 本轮必须写清目标、范围、非目标。
- 单节点、单目标、单轮交付优先。

## 17. 示例

### 17.1 本轮只做一致性评审，却误改 dashboard

场景：

- 本轮目标是“调整 `COLOR_CONSISTENCY_REVIEW` 的门禁规则”。
- 执行中顺手改了 dashboard 的统计卡片和待办排序。

为什么越界：

- dashboard 不属于本轮主目标。
- 即使评审门禁规则变化，dashboard 改动通常也不是本轮最小必要集。
- 这会引入新的页面验收面和新的回归风险。

正确处理：

- 打回 dashboard 改动。
- 若 dashboard 确实必须改，回到计划层重写范围与验收。

### 17.2 修一个 API bug 却顺带重构 auth

场景：

- 原始问题是某个 workflow API 在驳回时状态校验错误。
- 修复时顺手把 `auth` 模块的鉴权写法一起重构了。

为什么越界：

- `auth` 重构是独立目标，不是这个 bug 的最小修复。
- 它会扩大回归范围，增加权限风险和排查难度。
- 一旦验证失败，无法快速判断失败来自 bug 修复还是 auth 重构。

正确处理：

- 保留 workflow API 的最小修复。
- 将 auth 重构拆成独立任务。
