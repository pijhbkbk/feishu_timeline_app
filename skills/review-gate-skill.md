# Review Gate Skill

本技能面向 Review Gate Agent、Workflow Agent 和 ColorDev Orchestrator Agent，用于规范“轻卡新颜色开发项目管理系统”中驾驶室评审、一致性评审、目视色差评审等门禁型节点的设计、实现、记录、退回和验收方式。它只服务当前仓库的主链路与评审门禁语义，不服务通用审批系统抽象。

## 1. Skill 目标

- 为评审门禁型节点建立统一的设计、实现、验证与回滚规范。
- 强化“评审是门禁，不是备注”的系统约束，确保未通过不得进入后续主节点。
- 明确通过、驳回、退回、重提的处理原则，避免状态语义混乱。
- 确保评审历史、流转历史、审计日志完整保留，不覆盖旧记录。
- 约束 Review Gate Agent 与 Workflow Agent 在当前颜色开发项目语境内协同工作。

## 2. 适用范围

- 驾驶室评审 `CAB_REVIEW` 的规则设计与实现。
- 一致性评审 `COLOR_CONSISTENCY_REVIEW` 的规则设计与实现。
- 目视色差评审 `VISUAL_COLOR_DIFFERENCE_REVIEW` 的规则设计与实现。
- 新增评审门禁逻辑，例如补充材料、复评、会签要求。
- 调整评审通过、驳回、退回、重提的规则。
- 修复评审型缺陷，例如未通过仍放行、历史丢失、退回目标错误、重复评审记录。
- 审核评审节点对 `review_records`、`workflow_transitions`、`workflow_tasks`、`audit_logs`、`notifications` 的影响。

## 3. 不适用范围

- 无评审门禁语义的普通节点改造。
- 纯 UI 展示调整、纯文案修改、无状态变化的页面改造。
- 试图把当前系统改造成通用审批平台或通用 BPM 引擎。
- 与当前主链路无关的跨系统审批集成设计。
- 边界不清、不能收敛到单个评审目标的大范围自由改造。

## 4. 评审节点在当前系统中的角色

在当前系统中，评审节点不是附属说明节点，而是主链路上的放行门。它们承担的是“是否允许项目进入后续主节点”的业务裁决职责。

当前三类核心评审节点为：

| 评审节点 | 所在主链路位置 | 当前业务角色 |
| --- | --- | --- |
| `CAB_REVIEW` | `TRIAL_PRODUCTION` 之后 | 驾驶室评审门禁，决定是否可以继续向后推进。 |
| `COLOR_CONSISTENCY_REVIEW` | `CAB_REVIEW` 之后 | 一致性评审门禁，决定颜色一致性是否达标。 |
| `VISUAL_COLOR_DIFFERENCE_REVIEW` | `MASS_PRODUCTION` 之后 | 目视色差评审门禁，决定是否可以完成最终收尾。 |

它们共同承担以下角色：

- 把业务质量判断转化为可执行的后端流转裁决。
- 决定项目是进入后续主节点、退回整改，还是停留等待补充。
- 为后续审计、追责、复盘提供结构化评审证据。

## 5. 为什么“评审是门禁，不是备注”

因为在颜色开发项目里，评审结果直接决定主链路能否继续。如果把评审当成备注：

- 就会出现未通过仍进入后续主节点的错误放行。
- 就会让前端或人工习惯性绕过后端状态机。
- 就会使 `review_records` 失去门禁价值，沦为说明性文本。
- 就会使退回规则、通知策略、审计记录都失去一致性。

本项目必须坚持以下判断：

- 评审通过：允许进入定义好的后继节点。
- 评审驳回：必须进入明确的退回路径或整改路径。
- 评审补充：必须停留在门禁节点或其关联任务，不得伪装成放行。
- 评审重提：必须建立新的评审轮次或新的历史，不得覆盖旧结果。

未通过不得进入后续主节点，是本技能的硬性底线。

## 6. 三类评审节点的共同约束

- 评审结果必须由后端裁决，前端不能决定“已通过”或“允许推进”。
- 评审节点必须有明确的前置条件，不能在前序节点未完成时提前打开。
- 评审结果必须与 `review_records`、`workflow_transitions`、`workflow_tasks` 一起联动。
- 评审通过、驳回、退回、重提都必须保留完整历史，不允许覆盖旧记录。
- 评审相关关键写操作必须有权限校验、状态校验、数据范围校验。
- 评审相关关键写操作必须写入 `audit_logs`。
- 若评审动作触发通知，应在主业务成功后写入 `notifications` 并异步投递。
- 若评审规则变化影响主链路节点或退回路径，必须由 Workflow Agent 联动分析。
- 不允许为了扩展性提前抽象成通用审批平台。

## 7. 通过 / 驳回 / 退回 / 重提的处理原则

### 7.1 通过

- 通过表示当前门禁条件满足，可以进入后继主节点或触发后续并行节点。
- 后端必须校验当前节点、评审权限、表单完整性、必要附件与业务前置条件。
- 通过后必须新增 `review_records` 或完成当前评审记录，并写 `workflow_transitions`。
- 若通过后需要创建新任务或通知，必须同步处理 `workflow_tasks` 与 `notifications`。

### 7.2 驳回

- 驳回表示当前门禁未通过，必须进入定义好的退回目标。
- 驳回不是备注，也不是简单状态标记；它必须导致可追溯的后端流转结果。
- 驳回后必须写评审结论、退回原因、操作人、时间和目标节点。
- 驳回触发的退回路径必须显式定义，不能让前端推断。

### 7.3 退回

- 退回是驳回后的状态迁移动作，不等同于“评审结果字段变为 rejected”。
- 退回必须与主链路节点变化、任务关闭/重开、通知和审计一起考虑。
- 退回后的历史必须保留，不删除原任务、不覆盖原评审轮次。
- 退回目标节点错误会直接破坏主链路，必须纳入设计与测试重点。

### 7.4 重提

- 重提表示在驳回或补充材料后，再次提交新的评审轮次。
- 重提必须新增历史记录或新轮次标识，不能复用旧评审记录直接改结果。
- 重提前必须验证整改任务或补充材料是否满足再评审前置条件。
- 重提后必须重新进入评审门禁，而不是直接恢复为已通过。

## 8. 评审历史保留原则

- 历史不可覆盖，是评审技能的硬约束。
- 每次评审提交、驳回、补充、重提都应形成新的时间点记录。
- 旧 `review_records` 不得被原地覆盖为新的通过或驳回结果。
- `workflow_transitions` 必须保留每次评审触发的节点变化。
- `audit_logs` 必须保留谁在何时基于什么原因执行了什么评审动作。
- 若涉及附件补充，附件元数据应关联到当前评审轮次或动作上下文，而不是覆盖旧附件关系。
- 历史保留的目标不是“记录很多数据”，而是确保项目关闭前评审链条可追溯、可复盘、可审计。

## 9. review_records 与 workflow_transitions / audit_logs 的关系

| 对象 | 作用 | 与评审门禁的关系 |
| --- | --- | --- |
| `review_records` | 保存评审结论、意见、轮次、提交信息 | 是评审事实本身的核心记录。 |
| `workflow_transitions` | 保存节点迁移历史 | 是评审触发流程推进或退回的轨迹记录。 |
| `audit_logs` | 保存关键写操作审计信息 | 是“谁做了什么”层面的合规记录。 |

联动原则：

- 评审通过时，不能只写 `review_records` 而不写 `workflow_transitions`。
- 评审驳回并退回时，不能只改流程状态而不写评审记录。
- 关键评审动作不能只写业务表而不写 `audit_logs`。
- 这三类记录各自角色不同，不能互相替代。
- 若设计只覆盖其中一类记录，说明实现不完整。

## 10. 评审节点的前置条件与后继影响

### 10.1 前置条件

评审节点开启前必须明确以下内容：

- 当前项目确实处于允许进入该评审节点的主链路位置。
- 前序节点任务已完成，或明确满足进入评审的业务条件。
- 必填表单字段、必要附件、执行人角色都已满足。
- 若存在并行节点依赖，必须说明是否要求并行节点完成或仅要求主链路条件满足。
- 当前项目未处于关闭、撤销或非法状态。

### 10.2 后继影响

评审结果会影响以下内容：

- 主链路是否继续推进。
- 是否触发退回到特定节点。
- 是否重开或新建 `workflow_tasks`。
- 是否产生新的待评审提醒、退回提醒或整改提醒。
- 是否新增审计日志与附件关联上下文。

设计评审节点时，必须同时回答两个问题：

1. 评审未通过时，项目回到哪里，谁继续处理？
2. 评审通过时，项目进入哪里，是否需要新任务或并行任务？

## 11. 评审权限校验原则

- 评审权限必须由后端基于角色、数据范围和当前节点共同裁决。
- `reviewer` 不是万能权限；必须结合当前项目、当前节点和动作类型判定。
- `project_manager`、`process_engineer`、`quality_engineer` 等角色能否发起、补充、重提、查看，也必须显式定义。
- 无权限用户不得提交评审结论，不得伪造通过或驳回。
- 权限不足时，接口必须返回明确错误，不得静默失败或让前端自行回退。
- 任何会改变评审结果、流程节点、任务状态的动作都必须有权限校验与状态校验。
- 若本轮改动放宽了评审权限边界，必须进入中枢 Agent 的风险审查。

## 12. 评审表单与附件要求的约束写法

评审设计中，表单与附件约束应写成“门禁条件”，而不是模糊描述。推荐写法：

```markdown
## 表单要求
- 必填字段：
- 可选字段：
- 缺失时的拒绝策略：

## 附件要求
- 必需附件：
- 可选附件：
- 附件与当前评审轮次的关联方式：

## 提交校验
- 当前节点校验：
- 角色校验：
- 数据范围校验：
- 是否允许重提：
```

编写原则：

- 不要写“建议上传附件”，要写“缺失时是否允许提交”。
- 不要写“可填写说明”，要写“哪些字段是驳回/通过的必要条件”。
- 附件只存对象存储元数据，数据库只存关联信息。
- 附件要求变化若影响评审通过条件，必须纳入后端 Service 校验。

## 16. 常见错误与禁止事项

- 把评审当成备注字段，而不是门禁节点。
- 允许未通过评审继续进入后续主节点。
- 驳回时只写评审意见，不触发正确退回。
- 重提时覆盖旧评审记录。
- 前端直接决定是否通过、是否退回、是否可提交。
- 未做权限校验或只做前端按钮隐藏。
- 评审动作缺少 `audit_logs`。
- 评审动作影响了流程，但没有写 `workflow_transitions`。
- 为图方便把补充材料、驳回、退回混成同一状态。
- 修改评审退回目标时不分析主链路和并行节点影响。
- 把当前项目评审机制抽象成通用审批平台。

## 18. 示例

### 18.1 新增一致性评审逻辑

示例目标：在 `COLOR_CONSISTENCY_REVIEW` 增加“补充材料后复评”的门禁逻辑，但未补齐材料前不得进入 `MASS_PRODUCTION_PLAN`。

#### 设计要点

- 单一目标是增强一致性评审，不改动其它评审节点。
- 补充材料不是通过，也不是驳回；它应停留在当前评审门禁内。
- 补充材料请求需要新的 `review_records` 轮次或明确的动作记录，不能覆盖原评审结论。
- 当前节点仍是 `COLOR_CONSISTENCY_REVIEW`，直到后端确认复评通过，才可进入 `MASS_PRODUCTION_PLAN`。
- 若复评驳回，仍按现有规则退回 `PAINT_DEVELOPMENT`。

#### 验收重点

- 未补齐材料时不能放行到 `MASS_PRODUCTION_PLAN`。
- 补充材料请求与正式驳回在历史、任务、通知上必须可区分。
- 复评通过后才产生新的流转记录。

### 18.2 修改驳回后退回到哪个节点

示例目标：将某次规则调整中的 `CAB_REVIEW` 驳回目标从错误节点纠正回 `TRIAL_PRODUCTION`。

#### 设计要点

- 先确认当前正确业务语义仍然是 `CAB_REVIEW -> TRIAL_PRODUCTION`。
- 修改的不只是“目标枚举”，还包括退回后的 `workflow_tasks`、通知内容、审计描述和测试断言。
- 已存在的历史记录不能被回写修正；只能保证新发生的驳回按新规则执行。
- 若此前错误规则已影响其它节点，必须由中枢 Agent 决定是否做补偿或仅修正未来行为。

#### 验收重点

- 新驳回动作是否稳定退回 `TRIAL_PRODUCTION`。
- `review_records`、`workflow_transitions`、`audit_logs` 是否都反映正确的退回目标。
- 前端是否只展示后端返回结果，而不是本地写死退回目标。
