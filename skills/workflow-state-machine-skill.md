# Workflow State Machine Skill

本技能面向 ColorDev Orchestrator Agent、Workflow Agent、Review Gate Agent，用于约束“轻卡新颜色开发项目管理系统”中所有与 `workflow` 状态机相关的任务拆解、设计、实现和验收方式。它只服务当前仓库的主链路节点、并行节点、退回规则与后端裁决模型，不服务通用 BPM 理论扩展。

## 1. Skill 目标

- 为工作流状态机相关任务提供统一的分析、设计、编码与验收规范。
- 强化单节点、单目标、可验证、可回滚的最小可行执行原则。
- 确保主链路节点、并行节点、退回规则都由后端统一裁决，前端只消费结果。
- 确保状态变更与评审结果保留完整历史，覆盖 `workflow_tasks`、`workflow_transitions`、`review_records`、`audit_logs`。
- 明确本仓库只做颜色开发项目管理系统 MVP，不提前抽象通用 BPM 引擎。

## 2. 适用场景

- 新增主链路节点。
- 新增并行节点或调整并行节点触发时机。
- 修改退回规则、撤回规则、重提规则。
- 增加或调整评审门禁，影响 `review_records` 与节点推进关系。
- 修复状态流转错误、任务重复创建、历史缺失、通知重复触发等流程型缺陷。
- 调整与 `workflow_tasks`、`workflow_transitions`、`review_records` 相关的后端状态机实现。
- 为 ColorDev Orchestrator Agent 提供 workflow 设计与验收基线。

## 3. 不适用场景

- 无边界的全仓重构。
- 为未来通用性提前建设规则引擎、设计器或 BPM 平台。
- 与工作流状态机无关的纯 UI 美化、纯文案修改、纯报表展示。
- 跳过现有业务边界，直接让前端管理流程状态。
- 没有单一业务目标、无法收敛到单节点或单条规则的“大而全”改造。

## 4. 当前工作流边界与约束

- 项目目标是“轻卡新颜色开发项目管理系统”MVP，架构是模块化单体，不做微服务。
- 工作流流转、并行节点创建、退回逻辑、评审通过/驳回裁决必须由后端控制。
- 前端只负责展示、输入、交互状态，不得直接修改流程状态，不得在本地推导下一节点。
- 评审是门禁，不是备注。未通过评审不得进入后续主节点。
- 驳回、撤回、重提必须保留完整历史，不允许覆盖原 `review_records`、`workflow_tasks`、`workflow_transitions`。
- `schema.prisma` 是唯一数据模型源；若状态机变更牵动表结构，必须同步 migration。
- `audit_logs` 必须落库；状态机相关关键写操作不能只打控制台日志。
- `notifications` 是主业务成功后的异步机制，写入 `notifications` 并异步投递；队列失败不反向回滚主事务。
- 当前仓库只做主链路与明确的并行节点，不提前抽象通用 BPM 引擎。
- 本地环境若缺少真实 PostgreSQL / Redis，不能伪造联调结论，应以仓库可执行验证和明确风险说明替代。

## 5. 主链路节点说明

当前主链路节点如下，任何新增或调整都必须先说明它在主链路中的精确位置与门禁意义：

| 顺序 | 节点编码 | 中文节点 | 当前定位 |
| --- | --- | --- | --- |
| 1 | `PROJECT_INITIATION` | 项目立项 | 创建项目与流程实例的起点。 |
| 2 | `DEVELOPMENT_REPORT` | 新颜色开发报告 | 形成开发输入与基础信息。 |
| 3 | `PAINT_DEVELOPMENT` | 涂料开发 | 颜色开发主作业节点。 |
| 4 | `SAMPLE_COLOR_CONFIRMATION` | 样板确认 | 确认样板，后续可触发并行节点。 |
| 5 | `PAINT_PROCUREMENT` | 涂料采购 | 采购完成后触发性能试验与标准板等并行路径。 |
| 6 | `FIRST_UNIT_PRODUCTION_PLAN` | 首台计划 | 进入试制前的计划节点。 |
| 7 | `TRIAL_PRODUCTION` | 样车试制 | 驾驶室评审前的关键准备节点。 |
| 8 | `CAB_REVIEW` | 驾驶室评审 | 评审门禁节点，未通过不得继续主链路。 |
| 9 | `COLOR_CONSISTENCY_REVIEW` | 一致性评审 | 评审门禁节点，控制颜色一致性放行。 |
| 10 | `MASS_PRODUCTION_PLAN` | 排产计划 | 批量生产前计划节点。 |
| 11 | `MASS_PRODUCTION` | 批量生产 | 批产执行节点。 |
| 12 | `VISUAL_COLOR_DIFFERENCE_REVIEW` | 目视色差评审 | 评审门禁节点，控制色差放行。 |
| 13 | `PROJECT_CLOSED` | 颜色退出 | 项目关闭与颜色退出终点。 |

工作流相关任务必须先回答三个问题：

1. 变更发生在哪个主链路节点之前、之中或之后？
2. 它是否改变评审门禁或退回路径？
3. 它是否会影响当前项目的 `current node`、任务创建、历史记录或通知时机？

## 6. 并行节点处理原则

当前已定义的并行节点如下：

| 并行节点 | 触发条件 | 原则说明 |
| --- | --- | --- |
| `COLOR_NUMBERING` | `SAMPLE_COLOR_CONFIRMATION` 通过后创建 | 并行创建，不改变主链路推进裁决。 |
| `PERFORMANCE_TEST` | `PAINT_PROCUREMENT` 完成后创建 | 与标准板相关路径并行存在。 |
| `STANDARD_BOARD_PRODUCTION` | `PAINT_PROCUREMENT` 完成后创建 | 并行创建，但主链路推进条件必须明确。 |
| `BOARD_DETAIL_UPDATE` | `STANDARD_BOARD_PRODUCTION` 完成后自动创建 | 属于链式并行节点，不应由前端手动决定。 |
| `DEVELOPMENT_ACCEPTANCE` | `CAB_REVIEW` 通过后并行创建 | 驾驶室评审通过后派生的并行任务。 |

并行节点处理时必须遵守以下原则：

- 并行节点必须有明确的后端触发源，不能由前端页面“顺手创建”。
- 并行节点的创建条件、负责人、任务状态、通知时机必须可枚举。
- 并行节点完成与否是否影响主链路推进，必须写清楚，不能靠隐式约定。
- 并行节点若产生额外任务，必须落到 `workflow_tasks`，并写入必要的 `workflow_transitions` 或关联日志。
- 并行节点失败、撤回、重提时，必须说明是否影响主链路当前节点与评审门禁。
- 不允许为了支持未知未来场景，把并行节点抽象成宽泛的通用子流程引擎。

## 7. 退回规则处理原则

当前退回规则如下：

| 来源节点 | 退回目标 | 当前语义 |
| --- | --- | --- |
| `CAB_REVIEW` | `TRIAL_PRODUCTION` | 驾驶室评审未通过时，返回试制阶段整改。 |
| `COLOR_CONSISTENCY_REVIEW` | `PAINT_DEVELOPMENT` | 一致性评审未通过时，回到涂料开发整改。 |
| `VISUAL_COLOR_DIFFERENCE_REVIEW` | `MASS_PRODUCTION` | 目视色差评审未通过时，回到批产处理。 |

退回规则设计和实现必须遵守以下原则：

- 退回必须由后端基于当前节点、动作类型、角色权限、评审结果共同裁决。
- 退回路径必须显式定义来源节点与目标节点，禁止“就近回退”或前端自行猜测。
- 退回必须保留历史，不覆盖旧任务、不覆盖旧评审记录、不删除既有流转记录。
- 退回原因、操作人、时间、上下文说明必须可追溯，必要时同时写入 `workflow_transitions` 与 `audit_logs`。
- 撤回、驳回、重提是不同动作，不能混用同一状态语义。
- 若修改退回规则影响评审门禁或并行节点触发，必须连带分析相关节点的任务重建与通知影响。
- 若退回会跨越多个主链路节点，必须明确说明为何合理，以及是否破坏当前 MVP 的主链路语义。

## 8. 状态迁移建模原则

状态迁移建模必须贴合当前仓库，而不是抽象成通用 BPM 理论。建议按以下方式建模：

- 以单节点、单目标为最小分析单位，一次只讨论一个节点动作或一条规则变化。
- 将迁移动作建模为“当前节点 + 触发动作 + 前置条件 + 角色权限 + 评审结果 -> 新节点 / 新任务 / 新历史”。
- 主链路状态、任务状态、评审状态、通知状态要分层表达，避免一个字段承载全部语义。
- 节点流转规则可以代码化，节点定义可以数据化；但不要因此提前构建通用工作流引擎。
- 所有会改变项目当前节点、任务状态、评审结论的写操作都必须幂等。
- 每次合法迁移都应留下不可覆盖的 `workflow_transitions` 记录。
- 每次评审型迁移都应明确对应 `review_records` 的新增、关闭或补充行为。
- 若迁移会创建待办或新的执行责任，必须同步维护 `workflow_tasks`。
- 若迁移会引发提醒，通知应在主业务成功后写入 `notifications` 并异步投递。
- 每一条迁移都应能回答“如何回退”和“如何验收”两个问题。

## 9. 前后端职责分界

| 领域 | 前端职责 | 后端职责 |
| --- | --- | --- |
| 节点展示 | 展示当前节点、历史、任务状态、评审结果、错误提示。 | 返回当前节点、可执行动作、状态说明。 |
| 用户输入 | 收集用户动作、表单、附件、确认信息。 | 校验动作是否合法、校验当前状态与角色权限。 |
| 状态裁决 | 不允许前端决定是否流转、退回或通过。 | 决定能否迁移、迁移到哪里、是否创建并行节点。 |
| 历史记录 | 只展示历史。 | 写入 `workflow_transitions`、`review_records`、`audit_logs`。 |
| 通知 | 展示通知结果与发送状态。 | 决定何时写入 `notifications` 并触发异步投递。 |

强制规则：

- 前端不得直接修改流程状态。
- 前端不得本地拼接下一节点。
- 前端不得绕过 API 直接改变评审结果或任务状态。
- 任何流程裁决、评审门禁、退回路径都必须在后端 Service 层完成。

## 10. 与 review_records / workflow_transitions / workflow_tasks 的关系

| 对象 | 角色 | 必须保证的关系 |
| --- | --- | --- |
| `workflow_tasks` | 当前可执行任务与责任载体 | 节点推进、退回、重提、并行节点创建时，应正确创建、关闭或重开任务。 |
| `workflow_transitions` | 流转历史 | 每次主链路推进、退回、撤回、重提都必须保留历史，不可覆盖。 |
| `review_records` | 评审证据与门禁结果 | `CAB_REVIEW`、`COLOR_CONSISTENCY_REVIEW`、`VISUAL_COLOR_DIFFERENCE_REVIEW` 等评审节点必须有对应记录。 |

联动原则：

- 评审通过不能只改项目当前节点，必须同时记录评审结果与流转历史。
- 评审驳回不能只写 `review_records`，必须同步处理任务状态与退回路径。
- 撤回、重提必须新增历史记录或新任务，不得直接复写已有记录。
- `workflow_tasks` 负责“现在谁来做”，`workflow_transitions` 负责“流程怎么走过来的”，`review_records` 负责“门禁为什么放行或拦截”。
- 若状态机变更遗漏了这三类对象中的任何一类，说明设计或实现不完整。

## 11. 设计阶段检查清单

- [ ] 本轮目标是否收敛为单节点、单目标的工作流变更？
- [ ] 是否明确涉及主链路节点、并行节点还是退回规则？
- [ ] 受影响的来源节点、目标节点、前置条件、动作类型是否已枚举？
- [ ] 是否明确哪些角色能执行该动作，哪些角色只能查看？
- [ ] 是否明确评审是否为门禁，以及评审通过/驳回如何影响流转？
- [ ] 是否明确 `workflow_tasks`、`workflow_transitions`、`review_records` 的联动方式？
- [ ] 是否明确通知是否需要新增、变更或避免重复触发？
- [ ] 是否明确审计日志点位与落库内容？
- [ ] 若涉及数据模型变化，是否已判断需要更新 `schema.prisma` 与 migration？
- [ ] 是否有清晰的回滚点与失败回退路径？
- [ ] 是否明确前端只展示后端返回结果，而不是参与裁决？
- [ ] 是否避免把问题扩展为通用 BPM 设计？

## 12. 编码阶段检查清单

- [ ] 先实现后端规则，再实现前端展示或交互调整。
- [ ] Controller 只做协议适配，状态机规则写在 Service。
- [ ] 所有关键写操作都包含当前状态校验、权限校验、数据范围校验。
- [ ] 节点迁移实现具备幂等性，避免重复提交造成状态错乱。
- [ ] 状态变化后正确写入 `workflow_transitions`。
- [ ] 评审型流转正确写入或新增 `review_records`。
- [ ] 任务创建、关闭、重开、并行派生正确落到 `workflow_tasks`。
- [ ] 通知在主业务成功后写入 `notifications`，不让异步失败污染主事务结果。
- [ ] 审计日志写入数据库，不用控制台日志替代。
- [ ] 若有模型变化，`schema.prisma`、migration、代码引用保持一致。
- [ ] 前端只调用 API 并展示结果，不在本地维护流程状态机真值。
- [ ] 未经明确需求，不修改无关节点、无关规则、无关模块。

## 13. 测试与验收清单

功能与规则验收至少覆盖以下内容：

- [ ] 主链路正向推进是否符合设计。
- [ ] 并行节点是否在正确时机由后端创建。
- [ ] 退回规则是否只在合法条件下触发，并回到正确节点。
- [ ] 评审通过、驳回、撤回、重提是否保留完整历史。
- [ ] `workflow_tasks`、`workflow_transitions`、`review_records` 是否联动一致。
- [ ] 权限不足、状态不合法、重复提交时是否正确拒绝。
- [ ] `notifications` 是否按预期写入且无重复触发。
- [ ] `audit_logs` 是否覆盖关键状态变更。
- [ ] 前端是否无法绕过后端直接改流程状态。
- [ ] 回滚路径是否明确，失败后是否能恢复到最近稳定点。

仓库当前检查命令应以根目录 `AGENTS.md` 为准，最低检查链路为：

- `pnpm install`
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm --filter web build`
- `pnpm --filter api build`
- `pnpm --filter api prisma:validate`

环境限制处理原则：

- 若当前缺少真实 PostgreSQL / Redis，不能声称已完成真链路联调。
- 可接受的替代验证应明确写出，例如单元测试、集成测试、route smoke、`prisma:validate`、手工状态迁移检查。
- 缺少验证证据时，不得宣告状态机改动完成。

## 14. 常见错误与禁止事项

- 把流程流转判断写到前端。
- 把评审当备注处理，允许未通过评审继续推进主链路。
- 只更新项目当前节点，不写 `workflow_transitions`。
- 只更新评审结论，不处理 `workflow_tasks`。
- 为方便实现覆盖旧记录，而不是新增历史。
- 以“后续再抽象”为由先引入通用 BPM 结构。
- 忽略幂等性，导致重复点击产生重复任务或重复流转。
- 修改退回规则时不分析并行节点、副作用通知和权限变化。
- 让通知失败回滚主业务事务。
- 改了 `schema.prisma` 却没有 migration 或校验。
- 未经需求授权，顺手重排整个主链路。

## 16. 附一个“新增节点”示例

示例目标：在 `TRIAL_PRODUCTION` 与 `CAB_REVIEW` 之间新增 `TRIAL_RESULT_CONFIRMATION` 节点，用于确认试制结果后再进入驾驶室评审。

### 16.1 单一目标

- 只新增一个主链路节点。
- 不顺带改造其它评审节点。
- 不抽象通用子流程引擎。

### 16.2 设计要点

- 新节点位置必须明确为 `TRIAL_PRODUCTION -> TRIAL_RESULT_CONFIRMATION -> CAB_REVIEW`。
- 后端负责判断何时从 `TRIAL_PRODUCTION` 进入新节点，以及何时允许进入 `CAB_REVIEW`。
- 若新节点需要待办，必须新增对应 `workflow_tasks`。
- 每次进入、通过、退回都必须写 `workflow_transitions`。
- 若新节点本身不是评审门禁，就不要错误地写入 `review_records`；若它具有审批性质，则必须定义对应评审记录。
- 前端只能展示新节点状态和提交动作，不允许本地决定“试制结果已确认，所以可进入驾驶室评审”。

### 16.3 验收重点

- 正向流转是否变为三段式，而非直接跳到 `CAB_REVIEW`。
- 旧项目在无此节点数据时是否有明确兼容策略。
- 通知是否只在新任务创建或节点推进时触发一次。
- 历史记录是否完整，且可回滚到新增节点引入前的稳定方案。

## 17. 附一个“修改退回规则”示例

示例目标：调整 `COLOR_CONSISTENCY_REVIEW` 的退回处理，将“补充资料”与“评审驳回”区分开来。

### 17.1 规则定义

- 评审驳回：仍按当前规则退回 `PAINT_DEVELOPMENT`。
- 补充资料：不改变主链路当前节点，只在当前评审节点下创建新的补充任务。

### 17.2 设计要点

- 不能把“补充资料”错误实现成一次完整退回，否则会破坏主链路语义。
- `review_records` 必须区分驳回与补充资料请求，不能复写同一条记录。
- 真正退回到 `PAINT_DEVELOPMENT` 时，必须写新的 `workflow_transitions`，并更新相关 `workflow_tasks`。
- 仅补充资料时，应创建或重开任务，但不能伪造主链路回退历史。
- 通知策略也应区分两类动作，避免把补资料提醒当成节点退回提醒。

### 17.3 验收重点

- 驳回路径是否仍准确回到 `PAINT_DEVELOPMENT`。
- 补资料路径是否保持在 `COLOR_CONSISTENCY_REVIEW` 当前节点。
- 两类动作的历史、任务、通知、审计是否都能区分。
- 前端是否仍旧只展示后端裁决结果，而不是本地判断这是“退回”还是“补资料”。
