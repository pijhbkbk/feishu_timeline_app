# Bug Repro And Fix Skill

本技能面向 Bug Fix Agent、Regression Guard Agent、API Agent、Frontend Route Agent 和 ColorDev Orchestrator Agent，用于规范“轻卡新颜色开发项目管理系统”中 bug 的复现、最小定位、定向修复、回归验证和风险说明方式。它只服务当前仓库的 workflow、review、notification、audit、前后端合同等缺陷场景，不服务通用故障管理方法论。

本技能的四条硬约束：

- 先复现，再修复。
- 先补失败用例，再做最小修复。
- 修复后必须做完整回归。
- 禁止借修 bug 顺手大重构。

## 1. Skill 目标

- 为本仓库建立统一的缺陷复现与修复流程。
- 强化失败测试、最小修复、完整回归的执行纪律。
- 降低 workflow、review、notification、audit 等关键链路修复时的二次回归风险。
- 让每次 bug 修复都能留下清晰的风险说明与回滚依据。
- 约束 Bug Fix Agent 与其他专用 Agent 的协作边界。

## 2. 适用场景

- workflow 流转错误。
- review 通过/驳回/退回逻辑错误。
- notification 重复发送、漏发、错发。
- audit 缺失、记录错误、无法追溯。
- 前端页面展示与后端真实状态不一致。
- DTO、shared enums、错误响应导致的前后端合同缺陷。
- 权限、数据范围、幂等、状态校验导致的高风险写接口 bug。

## 3. 不适用场景

- 没有明确问题描述、无法观察到异常结果的“预防性大改”。
- 借修 bug 为名做架构重构、目录重排、通用引擎抽象。
- 与当前 bug 无关的样式优化、命名清洗、代码审美整理。
- 没有复现依据、没有验证目标的拍脑袋修复。
- 以“先让它别报错”为目标但不保证业务正确性的临时掩盖。

## 4. bug 修复的基本原则

- 先确认 bug 真实存在，再动代码。
- 先确定影响面，再决定修复范围。
- 修复必须围绕单个缺陷目标，尽量保持单节点、单目标、可验证、可回滚。
- 对高风险链路要先补失败用例，再做最小修复。
- 修复时尊重仓库边界：workflow、review、RBAC、audit、notification 的裁决逻辑仍留在后端。
- 修复完成后必须执行与影响面匹配的回归验证。
- 若发现问题根因超出本轮范围，应升级为计划重做，而不是继续扩写修复。

## 5. 为什么不能“直接改到不报错”

“不报错”不等于“修好了”，原因包括：

- 业务状态可能仍然错误，只是错误被隐藏了。
- review / workflow / audit / notification 往往是联动的，静默失败会制造更深的数据不一致。
- 前端不报错不代表后端规则正确。
- 节点退回、评审门禁、通知幂等等问题若只压制异常，很容易形成历史脏数据。

正确目标不是“看起来没报错”，而是：

- 复现路径消失。
- 失败用例转为通过。
- 关联回归没有被破坏。
- 风险边界被明确写清。

## 6. 先复现、再补失败用例、再最小修复的流程

标准流程如下：

1. 复现 bug  
   明确触发条件、输入、当前节点、角色、期望结果、实际结果。

2. 固化失败证据  
   记录日志、接口返回、状态变化、页面截图或可重放步骤。

3. 补失败用例  
   优先补最接近根因的测试、断言、route smoke 或最小复现实例。

4. 最小修复  
   只修改导致缺陷的必要代码，不扩散到无关模块。

5. 回归验证  
   验证原始问题已消失，并覆盖受影响链路的相邻场景。

6. 风险说明  
   写明修复边界、未覆盖风险、需要继续观察或后续任务的点。

流程纪律：

- 没复现，不进入修复。
- 没失败用例，不进入稳定修复宣告。
- 没回归，不进入完成结论。

## 7. 如何界定最小改动面

最小改动面指的是：

- 只修改真正导致缺陷的模块、函数、规则或页面。
- 不顺手重排目录、不顺手改命名、不顺手统一风格。
- 不把 unrelated DTO、枚举、查询、页面一起改掉。

界定方法：

- 先找出 bug 的触发链路。
- 确认根因落在哪一层：前端展示、API 协议、Service 规则、workflow 规则、review 门禁、notification 幂等、audit 埋点。
- 只修改根因所在层，以及因合同或联动必须同步的一小段代码。

若修复需要跨多个大模块，先问两个问题：

- 是真正的联动依赖，还是顺手想一起改？
- 本轮修复是否已经超出单个 bug 的可控边界？

## 8. 后端 bug、前端 bug、workflow bug、review bug、notification bug 的定位思路

### 8.1 后端 bug

优先检查：

- Controller 是否只做协议适配。
- Service 中的状态校验、权限校验、数据范围校验是否正确。
- DTO、返回结构、错误语义是否与页面合同一致。
- Prisma 查询或更新是否误写、漏写、越权写。

### 8.2 前端 bug

优先检查：

- 页面是否错误地本地拼状态。
- shared enums / 文案 / DTO 是否和后端不一致。
- 错误提示是否吞掉了真实业务错误。
- 页面是否误把交互状态当成业务真值。

### 8.3 workflow bug

优先检查：

- 当前节点与目标节点映射是否正确。
- `workflow_transitions` 是否按预期写入。
- 退回规则、并行节点触发条件、幂等保护是否失效。
- 是否有前端绕过后端直接改状态的路径。

### 8.4 review bug

优先检查：

- `review_records` 是否与 workflow 结果一致。
- 评审通过/驳回/重提是否保留历史。
- 未通过是否被错误放行到后续主节点。
- reviewer、project_manager 等角色边界是否被放宽。

### 8.5 notification bug

优先检查：

- 是否存在重复业务事件导致重复通知。
- 幂等键、去重逻辑、发送状态是否缺失。
- Redis 队列重试是否重复创建主通知记录。
- 业务成功与通知发送顺序是否错误。

## 9. 修复后必须验证的命令

修复后至少应按仓库当前规则执行以下验证链路：

- `pnpm install`
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm --filter web build`
- `pnpm --filter api build`
- `pnpm --filter api prisma:validate`

补充原则：

- 若 bug 只影响局部，也不能跳过与该链路强相关的验证。
- 若当前环境缺少真 PostgreSQL / Redis，必须明确说明验证边界，不伪造联调结论。

## 10. 风险回归说明的写法

风险说明应至少覆盖：

- 原始缺陷是什么。
- 根因定位在哪里。
- 本次修复改动了哪些模块。
- 哪些相邻链路已回归验证。
- 哪些风险仍未关闭。
- 若失败再次出现，最近回滚点是什么。

推荐模板：

```markdown
## 风险说明
- 原始问题：
- 根因：
- 最小修复范围：
- 已验证回归：
- 未关闭风险：
- 回滚点：
```

## 11. 设计检查清单

- [ ] 是否已拿到明确的复现步骤和实际结果？
- [ ] 是否明确期望结果与实际结果差异？
- [ ] 是否识别出 bug 属于前端、后端、workflow、review、notification、audit 中哪一类？
- [ ] 是否确定受影响模块和最小改动面？
- [ ] 是否决定需要补哪类失败用例？
- [ ] 是否确认修复不会越过仓库既有边界？
- [ ] 是否有清晰的回归验证范围？
- [ ] 是否写清了回滚点和未关闭风险？

## 12. 编码检查清单

- [ ] 已先补失败用例或最小复现实例。
- [ ] 修改只覆盖根因相关代码。
- [ ] 未借修 bug 顺手重构无关模块。
- [ ] workflow / review / RBAC / audit / notification 的边界没有被打破。
- [ ] 前端没有趁修 bug 接管业务裁决。
- [ ] 后端关键写接口的权限、状态、幂等校验未被绕开。
- [ ] 若影响通知或审计，相关记录链路已同步修正。
- [ ] 若影响 shared DTO / enums，前后端消费方已同步。

## 13. 验证检查清单

- [ ] 原始复现步骤已经失效，bug 不再出现。
- [ ] 新增失败用例已转为通过。
- [ ] 与 bug 相邻的 workflow / review / notification / audit 场景已回归。
- [ ] 没有引入新的权限、状态、数据范围问题。
- [ ] 前端展示与后端真实状态保持一致。
- [ ] 通知、审计、流转记录在需要的场景下完整可追溯。
- [ ] 规定检查命令已执行，或明确说明受环境限制的部分。

## 14. 常见错误与禁止事项

- 没复现就开始改代码。
- 没失败用例就宣称修复完成。
- 只修页面报错，不修后端根因。
- 为了“看起来好了”吞掉异常或绕开校验。
- 借修 bug 顺手大重构。
- 一次修多个无关缺陷，却没有清晰边界。
- workflow bug 用前端本地状态兜底。
- review bug 通过放宽权限或跳过门禁来“修好”。
- notification bug 通过删除去重逻辑或直接静默失败来“止血”。
- 修完不做回归，只测一条 happy path。

## 16. 示例

### 16.1 一致性评审驳回后流转错误

示例问题：

- `COLOR_CONSISTENCY_REVIEW` 驳回后，项目错误地进入 `MASS_PRODUCTION_PLAN`，而不是退回 `PAINT_DEVELOPMENT`。

正确修复方式：

- 先复现驳回动作下的错误流转。
- 补失败用例，断言驳回后应写入 `workflow_transitions` 且目标节点为 `PAINT_DEVELOPMENT`。
- 最小修复后端 workflow / review 联动逻辑。
- 回归验证：
  - 驳回路径
  - 通过路径
  - 审计日志与通知是否仍正确

### 16.2 重复通知 bug

示例问题：

- 用户重复提交一次评审动作，系统产生两条相同待评审通知。

正确修复方式：

- 复现重复提交或重试场景。
- 补失败用例，断言同一业务事件只生成一条有效通知。
- 最小修复通知幂等键或去重逻辑，不动无关页面。
- 回归验证：
  - 首次发送
  - 重复提交
  - 队列重试

### 16.3 错误示例：为修一个 bug 顺手改多个无关模块

错误做法：

- 原始问题只是一个评审退回 bug。
- 修复时顺手改了页面结构、DTO 命名、多个 shared 枚举、通知文案和无关查询逻辑。

为什么错误：

- 改动面失控，难以回滚。
- 无法判断到底哪一处是真正修复。
- 回归成本暴涨，且极易引入新的无关缺陷。

正确结论：

- bug 修复必须围绕根因做最小改动。
- 任何超出当前缺陷边界的改造，都应拆到独立任务中处理。
