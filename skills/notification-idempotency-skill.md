# Notification Idempotency Skill

本技能面向 Notification Agent、API Agent、Workflow Agent 和 ColorDev Orchestrator Agent，用于规范“轻卡新颜色开发项目管理系统”中站内通知、Redis 队列、mock 飞书消息适配层相关的触发、去重、重试、失败补偿和业务边界。它只服务当前仓库的通知中心设计，不服务通用消息平台建设。

本技能的四条底线：

- 通知逻辑不能反向裁决业务状态。
- 站内通知是当前 MVP 的事实落点，必须先写入 `notifications`。
- Redis 队列只负责异步投递与重试，不参与主事务成败。
- 通知必须做幂等与去重，避免重复提醒、重复待办、重复推送。

## 1. Skill 目标

- 为本仓库通知模块建立可执行的幂等、去重、重试和补偿规范。
- 明确 `notifications`、Redis 队列和 mock 飞书消息适配层的职责边界。
- 约束 workflow、review、audit 相关业务事件如何触发通知。
- 确保通知失败不会污染主业务结果，但通知链路仍可追踪、可排错。
- 降低重复提交、队列重试、节点回退等场景下的重复提醒风险。

## 2. 适用场景

- 新任务分配提醒。
- 待评审提醒。
- 节点退回提醒。
- 超期扫描提醒。
- 新增基于主链路节点或评审结果触发的站内通知。
- 为现有通知补充幂等键、去重规则、投递状态或重试策略。
- 修复重复提醒、重复待办、重复飞书推送、通知漏发等问题。
- 审核 workflow / review / API 改动是否会对通知链路产生副作用。

## 3. 不适用场景

- 纯前端消息样式改动。
- 不产生业务事件的只读查询。
- 不改变 `notifications`、队列投递或外部适配层行为的纯文案调整。
- 把当前系统扩展成通用消息平台、事件总线或多通道营销平台。
- 没有明确业务触发源的“先发通知再说”式设计。

## 4. 当前系统通知模块的角色

当前系统的通知中心角色已经明确：

- 站内通知是 MVP 必做，写入 `notifications`。
- Redis 队列只做异步投递与重试。
- 飞书消息当前为 mock 适配层，保留未来替换口。

因此，通知模块在当前系统中主要承担三件事：

- 向系统内用户提供待办和状态提醒。
- 将主业务成功后的通知事件异步投递到外部适配层。
- 为排错提供通知事实、发送状态和重试状态记录。

通知模块不是：

- workflow 裁决器。
- review 门禁执行器。
- 主事务成功与否的决定者。

## 5. `notifications` 与 workflow / review / audit 的关系

| 模块 | 回答的问题 | 与通知的关系 |
| --- | --- | --- |
| `workflow_transitions` | 流程从哪里到哪里 | 某些通知由流程流转触发，但流转记录不等于通知记录。 |
| `review_records` | 评审为什么通过或驳回 | 待评审、评审结果通知通常由评审事件触发。 |
| `audit_logs` | 谁在何时做了什么 | 用于排查是谁触发了通知相关业务动作。 |
| `notifications` | 给谁发了什么通知，状态怎样 | 是通知事实与投递状态的记录。 |

联动原则：

- workflow 或 review 变化可以触发通知，但通知不能代替 workflow / review 记录。
- `notifications` 应该能回溯到触发它的业务对象，例如项目、任务、评审、节点。
- `audit_logs` 应能帮助回答“是谁触发了这条通知对应的业务动作”。

## 6. 什么时候应该发通知，什么时候不该发

### 6.1 应该发通知

以下场景通常应该发通知：

- 新任务分配给具体负责人。
- 某评审节点进入待评审状态。
- 节点被驳回并退回到具体责任人处理。
- 超期扫描发现待办逾期。
- 项目进入需要明确责任人关注的新阶段，且已有业务定义要求提醒。

### 6.2 不该发通知

以下场景通常不应发通知：

- 只读查询。
- 草稿保存但未形成业务状态变化。
- 幂等重复请求命中，没有产生新的业务结果。
- 因状态不合法或权限不足而被拒绝的请求。
- 单纯为了“提醒一下”但没有明确责任人、明确对象、明确动作来源的事件。

判断原则：

- 只有当业务事实发生变化，且确实需要通知某个责任对象时，才应该发。
- 没有新业务事实，就不应生成新通知。

## 7. 站内通知与飞书适配层的分层原则

当前系统必须按三层处理通知：

### 7.1 站内通知层

- 以 `notifications` 表为事实源。
- 记录通知对象、接收人、业务关联、发送状态、重试状态。
- 前端待办、消息列表优先基于此层展示。

### 7.2 队列层

- 使用 Redis 队列异步投递。
- 负责重试、延迟、失败补偿触发。
- 不负责决定主业务成败。

### 7.3 外部适配层

- 当前为 mock 飞书消息适配层。
- 负责把站内通知或投递事件转换成外部通道调用。
- 不拥有业务裁决权。

分层结论：

- 站内通知先成立，再异步投递。
- 队列失败不等于业务失败。
- mock 飞书适配层失败不应回滚主业务。

## 8. 幂等键 / 去重策略 / 重试策略建议

### 8.1 幂等键建议

通知幂等键建议至少包含以下维度：

- 事件类型，例如 `task_assigned`、`review_pending`、`node_returned`
- 项目 ID
- 业务对象 ID，例如任务 ID、评审记录 ID、流转记录 ID
- 接收人 ID
- 轮次或版本信息，例如评审轮次、流转序号

参考模板：

```text
<event_type>:<project_id>:<object_id>:<receiver_id>:<round_or_transition_id>
```

### 8.2 去重策略建议

- 同一业务事件对同一接收人在同一轮次只生成一条有效通知。
- 重复提交若未产生新业务结果，不应生成新通知。
- 同一退回事件不应因为前端重试或队列重试而写入多条站内通知。
- 待评审提醒与超期提醒属于不同事件类型，不应混用同一幂等键。

### 8.3 重试策略建议

- 仅对异步投递失败做重试，不对主业务事务做“通知式重试”。
- 重试前应基于幂等键确认是否已经成功处理。
- 重试应区分：
  - 站内通知写入失败
  - 队列 enqueue 失败
  - 外部 mock 飞书适配层发送失败
- 对外部适配层失败，可以重试发送状态，但不能重复创建站内通知事实。

## 9. 业务成功与通知发送的先后关系

当前系统必须遵循以下顺序：

1. 主业务成功提交。
2. 写入 `notifications` 事实记录。
3. 再 enqueue 到 Redis 队列。
4. 再由 mock 飞书适配层异步处理外部发送。

原因：

- 通知是业务成功后的副产物。
- 先写站内通知，才能保证至少有系统内可见事实。
- 先 enqueue 或先发外部消息，会在主业务失败时制造伪通知。

## 10. 通知失败时是否影响主业务的原则

结论非常明确：

- 不影响主业务。

当前仓库规则已明确：

- 主业务成功后再 enqueue。
- 队列失败不回滚业务。
- 外部 mock 飞书消息失败也不回滚业务。

但这不代表通知失败可以忽略：

- 失败状态必须可记录。
- 重试链路必须可追踪。
- 排错时必须能分辨是业务没触发通知，还是通知投递失败。

## 11. 避免重复提醒、重复待办、重复推送的规则

- 相同幂等键的通知只允许一条有效事实记录。
- 相同业务事件不能因接口重复提交生成多条待办。
- 队列重试只能重试投递，不得重复创建通知主记录。
- 评审待办与评审结果通知应区分事件类型，避免相互覆盖或互相重复。
- 退回通知应绑定到具体退回事件或具体 `workflow_transition`，不能只按节点名粗暴生成。
- 超期提醒可按扫描周期重复，但必须有时间窗口和周期去重规则。
- 通知展示层若按未读统计或待办聚合，也不能因为数据库重复记录而放大错误。

## 15. 常见错误与禁止事项

- 先发通知，再提交主业务。
- 把通知是否成功发送当成主业务是否成功的判断条件。
- 重复提交导致重复写入 `notifications`。
- 队列重试时再次创建站内通知，而不是只重试投递。
- 没有幂等键，只靠“感觉不会重复”。
- 用节点名或文案粗暴去重，导致不同业务事件互相覆盖。
- 把 mock 飞书适配层做成业务裁决入口。
- 通知发送失败后静默吞掉，无法排错。
- 待评审提醒、退回提醒、超期提醒混用同一事件类型。
- 为提醒而提醒，产生没有责任人、没有动作闭环的噪声消息。

## 17. 示例

### 17.1 评审待办通知

示例场景：项目进入 `COLOR_CONSISTENCY_REVIEW`，需要给当前 reviewer 发待评审提醒。

正确做法：

- 以“进入待评审状态”作为业务触发源，而不是用户打开页面时触发。
- 先写一条 `notifications` 记录，关联项目、评审节点、reviewer。
- 使用幂等键，例如：
  - `review_pending:<project_id>:<review_task_id>:<reviewer_id>:<round>`
- 再 enqueue 到 Redis 队列，由 mock 飞书适配层异步发送。

必须避免：

- 页面刷新就重复发通知。
- reviewer 重复分配同一轮次时写多条完全相同待办。

### 17.2 退回节点通知

示例场景：`CAB_REVIEW` 驳回并退回 `TRIAL_PRODUCTION`，需要通知试制责任人。

正确做法：

- 触发源是具体退回动作或具体 `workflow_transition`，而不是“项目还在这个节点”。
- 幂等键建议绑定退回事件，例如：
  - `node_returned:<project_id>:<transition_id>:<receiver_id>`
- 通知内容与接收人基于这次退回事实生成。
- 若队列或外部发送失败，只重试投递，不重建退回通知主记录。

必须避免：

- 同一退回事件因重复点击或重试生成多条退回提醒。
- 仅用节点名去重，导致不同退回事件被错误合并。

### 17.3 错误示例：重复提交导致重复发信

错误做法：

- 用户连续点击“提交评审”两次。
- 后端每次都新建一条 `notifications` 记录。
- Redis 队列各自 enqueue。
- mock 飞书适配层收到两次相同发送任务。

为什么错误：

- 没有把业务幂等和通知幂等串起来。
- 没有用事件级幂等键约束同一业务事实只生成一条通知。
- 会制造重复待办、重复站内消息、重复外部提醒。

正确结论：

- 同一业务事实只能生成一条通知事实记录。
- 后续重复请求应命中幂等，不再重复建通知主记录，只保留可排查的重复命中语义。
