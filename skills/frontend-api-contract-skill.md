# Frontend-API Contract Skill

本技能面向 Frontend Route Agent、API Agent 和 ColorDev Orchestrator Agent，用于规范“轻卡新颜色开发项目管理系统”中页面字段、DTO、枚举、状态文案、接口协议、错误提示以及前后端职责分界。它只服务当前仓库的 Next.js 前端、NestJS 后端和 `packages/shared` 共享契约，不服务通用前后端协作理论。

本技能的四条硬约束：

- 前端只负责展示、输入、交互状态。
- 所有写操作必须走后端 API。
- 前端不能拼流程状态。
- shared enums / 状态文案 / DTO 必须保持一致。

## 1. Skill 目标

- 为本仓库建立统一、可执行的前端与 API 合同规范。
- 明确页面字段、DTO、枚举、状态文案在前后端之间的来源与边界。
- 避免前端本地拼接 workflow 状态、review 结论或任务状态。
- 统一错误响应与前端提示的处理方式。
- 约束附件上传、详情展示、动作提交等典型交互的前后端分工。

## 2. 适用场景

- 新增页面表单、新增详情页、新增业务视图。
- 新增或调整 DTO、接口出参、共享枚举、状态文案。
- 新增 workflow / review / task 相关的前端展示与交互。
- 新增写接口，例如提交、审批、退回、关闭、上传结果确认。
- 修复前后端字段不一致、枚举不一致、错误提示不一致的问题。
- 组织 Frontend Route Agent 与 API Agent 共同推进复杂页面改动。

## 3. 不适用场景

- 不涉及接口协议的纯样式、布局、美术优化。
- 仅修改后端内部实现，且不改变 DTO、错误响应或页面消费方式。
- 只修改数据库结构而不影响前端契约的纯 Prisma 迁移任务。
- 试图把当前项目抽象成通用前后端平台规范。
- 让前端承担 workflow、review、RBAC、audit 的业务裁决责任。

## 4. 前端与 API 的职责边界

前端职责：

- 展示项目、流程、评审、任务、附件和日志信息。
- 收集用户输入。
- 管理局部交互状态，例如 loading、禁用、错误提示、成功反馈。
- 根据后端返回的数据做页面布局、列表聚合、详情渲染和按钮显隐。

API 职责：

- 提供稳定 DTO、枚举、状态文案来源或状态码语义。
- 校验所有写动作的权限、状态、数据范围和业务规则。
- 决定 workflow 是否推进、review 是否通过、任务是否创建、通知是否触发。
- 返回可渲染、可提示、可追溯的数据结构。

结论：

- 前端不裁决流程。
- API 不依赖前端帮它保证规则正确。
- 页面是消费者，不是业务真值源。

## 5. 为什么前端不能拼流程状态

当前仓库已经明确：

- 流程流转必须由后端控制。
- 所有写操作必须走后端 API。
- 评审通过/驳回规则必须放在后端。

因此前端不能拼流程状态，原因包括：

- 前端看不到完整权限、数据范围和后端约束。
- workflow、review、RBAC、notifications、audit 通常同时联动，前端无法安全裁决。
- 前端本地推断“下一节点是什么”会制造状态漂移。
- 前端如果直接写阶段状态，会绕过审计、退回规则、幂等和权限检查。

正确做法：

- 前端只展示后端返回的当前节点、可执行动作、状态文案、历史记录。
- 用户点击动作后，提交“动作请求”，而不是提交“我想把状态改成 X”。

## 6. `packages/shared` 的复用原则

`packages/shared` 是共享类型、枚举、常量的优先来源。复用原则如下：

- 表单字段枚举、状态枚举、状态文案常量优先复用 shared。
- 不要在前端和后端各自复制一份相同枚举。
- 不要在页面里写死后端已经存在的状态字符串。
- 共享类型用于表达稳定契约，不用于承载前端局部 UI 状态。

适合放在 shared 的内容：

- workflow 节点枚举。
- review 类型枚举。
- 状态值常量。
- 稳定 DTO 结构或其核心字段类型。
- 通用错误码或稳定业务动作名。

不适合放在 shared 的内容：

- 页面本地临时状态。
- 纯前端样式文案。
- 只在单个服务内部使用且不对前端暴露的细节类型。

## 7. 表单字段、DTO、枚举、状态文案的一致性原则

### 7.1 表单字段

- 表单字段名应与 DTO 字段名保持语义一致。
- 页面字段顺序和展示方式可以变化，但字段含义不能变化。
- 必填、可选、默认值、只读态应以前后端合同为准，不靠页面猜测。

### 7.2 DTO

- DTO 是前后端合同，不是页面随意拼装的 JSON 习惯。
- 入参 DTO 应表达动作需要的输入。
- 出参 DTO 应表达页面渲染需要的事实，不暴露无必要内部结构。
- 新增字段时要同时检查前端消费和后端返回是否一致。

### 7.3 枚举

- workflow 节点、评审类型、通知类型、状态值等稳定集合应统一复用 shared enums。
- 不允许前端自己补一个“接近”的枚举映射。
- 若后端新增枚举值，前端渲染分支必须同步评估。

### 7.4 状态文案

- 状态值和状态文案要区分。
- 状态值用于逻辑判断。
- 状态文案用于展示。
- 如果 shared 已提供文案映射，前端应优先复用。
- 不要在多个页面手写不同版本的同一状态文案。

## 8. 写接口的动作化设计原则

写接口应以“动作”为中心，而不是以“直接改字段”为中心。

推荐设计：

- `submitProjectInitiation`
- `approveCabReview`
- `rejectColorConsistencyReview`
- `returnWorkflowNode`
- `closeProject`
- `bindAttachment`

不推荐设计：

- `updateStatus`
- `setCurrentNode`
- `patchReviewResult`

原因：

- 动作化接口更容易承载权限、状态、审计、通知和幂等规则。
- 动作化接口能避免前端直接传“我要把状态改成什么”。
- 动作化接口更贴合当前项目的 workflow / review 业务语义。

## 9. 页面与详情视图如何消费 workflow / review / task 数据

页面和详情视图应把 workflow、review、task 当作后端事实来消费。

推荐消费方式：

- 项目详情页展示：
  - 当前主链路节点
  - `workflow_transitions` 历史
  - `review_records` 历史
  - 当前待办任务或 `workflow_tasks`
- 列表页展示：
  - 当前节点摘要
  - 当前责任人
  - 当前状态文案
  - 是否存在待评审、退回、超期等标识
- 表单页展示：
  - 当前节点上下文
  - 可执行动作
  - 当前用户是否可提交

不推荐消费方式：

- 前端根据几个字段自行推断完整流程状态。
- 前端根据“上一个节点是什么”来猜“下一步应该是什么”。
- 前端自己拼出 review 和 workflow 的复合真值。

## 10. 错误响应与前端提示的处理原则

后端应返回稳定、可区分的错误语义，前端负责把它们转化成可理解提示。

建议至少区分：

- 权限不足
- 数据范围不足
- 当前状态不允许
- 参数校验失败
- 业务规则冲突
- 系统错误

前端处理原则：

- 不要自己猜错误类型。
- 优先根据后端错误码或稳定语义展示提示。
- 对高风险动作失败，应保留用户可理解的业务提示，而不是只显示技术异常。
- 页面应避免把任何失败都统一成“提交失败”。

## 11. 上传附件时前端与后端的职责分工

当前仓库边界已经明确：

- 上传文件只拿对象存储凭证或上传结果，不直接持久化业务关系。
- 附件只存对象存储，数据库只存元数据。

分工如下：

前端负责：

- 选择文件。
- 获取上传凭证或上传目标。
- 执行文件上传。
- 提交上传结果或对象存储 key 给后端。

后端负责：

- 校验上传结果是否合法。
- 将附件元数据落库。
- 将附件与项目、评审、流程对象建立业务关联。
- 写审计日志和必要的业务校验。

结论：

- 前端不能直接认为“文件传上去就等于附件绑定成功”。
- 业务关系的持久化必须由后端确认。

## 15. 常见错误与禁止事项

- 前端直接写阶段状态。
- 页面本地拼接“下一节点是什么”。
- 前后端各自维护一份状态枚举或状态文案。
- DTO 改了，但页面字段或后端返回没同步。
- 把 `edit`、`submit`、`approve` 混成一个写接口。
- 前端上传文件后直接把业务对象标记为已绑定附件。
- 把所有错误都展示成“请求失败，请稍后再试”。
- 让页面根据角色自行猜业务可执行动作，而不是消费后端结果。
- 把核心 workflow / review 规则写进 React 页面逻辑。

## 17. 示例

### 17.1 新增一致性评审表单

示例目标：新增 `COLOR_CONSISTENCY_REVIEW` 的前端表单与提交接口对接。

正确做法：

- 表单字段与后端评审提交 DTO 一一对应。
- 评审类型、评审结果候选值、状态文案优先复用 `packages/shared`。
- 提交时调用动作化 API，例如 `submitColorConsistencyReview` 或等价语义接口。
- 页面只提交动作和输入，不提交“把项目推进到 `MASS_PRODUCTION_PLAN`”这样的状态指令。

必须避免：

- 在前端根据“评审通过”直接把当前节点改为下一节点。
- 在页面内手写一套 review status 枚举。

### 17.2 详情页展示 `review_records` 与 `workflow_transitions`

示例目标：项目详情页同时展示评审历史和流程流转历史。

正确做法：

- 后端返回结构化的 `review_records` 和 `workflow_transitions` 数据。
- 前端负责按时间线或分区渲染。
- 状态文案、节点名和评审类型映射优先使用 shared 常量或后端稳定字段。
- 页面把这两类记录当作不同数据源展示，不混成一条模糊状态。

必须避免：

- 前端根据 `review_records` 自己推导完整 workflow 历史。
- 用单一“状态时间线”覆盖掉 review 和 workflow 的不同语义。

### 17.3 错误示例：前端直接写阶段状态

错误做法：

- 页面在用户点击“通过评审”后，本地直接把 `currentNode` 改成下一节点。
- 然后再发一个普通保存接口，试图让后端“顺便跟上”。

为什么错误：

- 绕过了后端 workflow 裁决、权限校验、状态校验和审计日志。
- 会制造前端显示与后端真实状态不一致。
- 无法保证 review、task、notifications、audit 的联动正确。

正确结论：

- 前端只能提交动作请求。
- 当前节点、评审结果、任务变化、通知触发、审计记录都必须以后端返回为准。
